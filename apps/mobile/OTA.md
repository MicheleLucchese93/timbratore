# sonoQui OTA updates

How to ship JavaScript/TypeScript changes to the sonoQui mobile app without a
TestFlight / Play Store review.

Server / keys / env setup lives in [`../../ota/README.md`](../../ota/README.md)
(the operator manual). This file is the **developer runbook**.

---

## How it works

```
  edit apps/mobile/src/…  ──►  apps/mobile/scripts/publish-ota.sh
                                          │
                          ┌───────────────┴────────────────┐
                          │  fingerprint gate               │
                          │  @expo/fingerprint vs           │
                          │  .native-fingerprint.json       │
                          └───────────────┬────────────────┘
                          match │                 │ differ
                                ▼                 ▼
                       eoas publish        ❌ refuse — native
                       --branch <…>           rebuild required
                                │
                                ▼
                  https://ota.sonoqui.pro  (expo-open-ota)
                                │
                                ▼  on next app launch / foreground
                       expo-updates checks /api/manifest,
                       downloads bundle, verifies signature,
                       reloads into the new JS (useUpdateGate)
```

### runtimeVersion is a static string

`app.json > runtimeVersion` is a hand-pinned string (currently `"1"`), **not**
the `fingerprint` policy. A static runtimeVersion guarantees the value the
`eoas publish` step stamps onto a bundle is identical to the value the binary
reports — there is no `@expo/fingerprint` recomputation that could drift between
your machine and the build (which is exactly the drift that bites self-hosted
OTA). The binary asks `ota.sonoqui.pro/api/manifest` for an update matching its
runtimeVersion; the server only serves bundles tagged with the same one.

**Bump `runtimeVersion` on every native change** (see "Native rebuilds" below).
The fingerprint gate is the safety net that reminds you to.

### The fingerprint gate

`scripts/publish-ota.sh` computes `@expo/fingerprint` for the project and
compares it to `.native-fingerprint.json`. If they match, the change is JS-only
and OTA-safe. If they differ, the native layer changed — publishing would push a
bundle that the shipped binary can't run — so the script refuses (override with
`--force` only if you are certain the divergence is spurious).

### Channels

The binary bakes an `expo-channel-name` header at build time
(`app.config.ts` reads `OTA_CHANNEL`; `eas.json` sets it per profile; hand-built
Xcode/Android archives default to `production`). The server maps channel →
same-named branch.

- `staging` — TestFlight / Play Internal
- `production` — App Store / Play Store production

### Code signing

Bundles are RSA-signed with `credentials/private-key.pem` and verified by the
app against `credentials/certificate.pem` (bundled into the binary). A stolen
server can't push a malicious update without the private key, and a hijacked
CF/DNS can't substitute a bundle without breaking the signature.

## Daily operations

### Ship a JS/TS fix (the happy path)

```bash
cd apps/mobile
npm run ota:publish            # → staging (TestFlight / Play Internal)
```

Devices on a `staging` binary pick it up on the next cold launch (or sooner via
the in-app `useUpdateGate` foreground check). Promote to production once happy:

```bash
npm run ota:publish:prod       # → production (App Store / Play prod)
```

Prerequisites (one-time): be logged into Expo (`npx eas login`) as
`micheel93-2`, and have `credentials/private-key.pem` present locally.

### Rollbacks

Server-side, via the dashboard (not CI):

1. Open https://ota.sonoqui.pro/dashboard (password = `OTA_ADMIN_PASSWORD` from
   `/opt/sonoqui/.env`).
2. Find the broken branch's latest release, mark it rolled back.

On next launch, devices fall back to the previously cached bundle (or the
binary's embedded bundle on a fresh install). Rollback is reversible from the
same dashboard.

## Native rebuilds (the not-so-happy path)

A "native rebuild" = a fresh binary in Xcode / Android Studio → a new
TestFlight / Play submission. OTA can't bridge these because the native layer
changed. Triggers: any dep bump (incl. `expo-*`), Expo SDK upgrade, a config
plugin add/remove/change, permissions, splash/icon/scheme/bundle-id, or direct
`ios/`/`android/` edits.

> **sonoQui is a bare-but-prebuild-driven Expo project.** `ios/` and `android/`
> are committed, but they are regenerated by `npm run prebuild` (which is
> `expo prebuild --clean`). The OTA config (updates URL, code-signing cert,
> channel header, runtimeVersion) lives in `app.json` + `app.config.ts` and is
> the source of truth — **`prebuild` wires it into the native projects.** Always
> run `npm run prebuild` before a native build so the binary is OTA-enabled.

Steps:

```bash
cd apps/mobile

# 1. Bump the static runtime version so old binaries don't pull an
#    incompatible bundle:
#    edit app.json  →  "runtimeVersion": "1"  →  "2"

# 2. Regenerate native projects with the OTA config baked in:
OTA_CHANNEL=staging npm run prebuild          # staging / TestFlight audience
#   …or OTA_CHANNEL=production for store builds (the app.config default is
#   already 'production', so a hand-archived Xcode build lands on production).

# 3. Build + submit:
cd ios && pod install && cd ..                # then archive in Xcode
npm run build:aab                             # Android AAB

# 4. Once the new binary is live on a device and boots cleanly, re-baseline
#    the fingerprint gate so future JS-only pushes are recognised as OTA-safe:
npm run ota:seed-fingerprint
git add apps/mobile/.native-fingerprint.json app.json
git commit -m "chore(mobile): bump OTA runtimeVersion + re-seed fingerprint baseline"
```

> ⚠️ **First OTA-enabled build:** the committed `.native-fingerprint.json` was
> captured *before* expo-updates was wired into the native projects, so your
> first `prebuild` will flip the fingerprint. That is expected — re-seed (step 4)
> after that first build and the gate settles.

> ⚠️ **`prebuild --clean` drifts manual native edits — re-apply after every run.**
> These are hand-maintained and NOT reproduced by any config plugin, so
> `expo prebuild --clean` silently reverts them. After prebuilding, restore:
> | File | Re-apply |
> |---|---|
> | `ios/Podfile` | `post_install` block clearing `INSTALL_OWNER`/`INSTALL_GROUP` on `installer.pods_project` (AD-joined-Mac chown fix) |
> | `ios/sonoQui/sonoQui.entitlements` | `aps-environment` → `production` (prebuild writes `development`) |
> | `android/app/src/main/res/values/colors.xml` | `colorPrimary` → `#15569e` (prebuild writes `#023c69`) |
> The `git checkout HEAD -- <file>` trick restores the Podfile + entitlements
> wholesale (their only diff vs HEAD is the manual edit). Consider promoting
> these to config plugins so they survive prebuild on their own.

> 📌 **Channel baked for this archive:** the native tree was prebuilt with the
> default channel `production`. For a **TestFlight build on the `staging`
> channel**, flip the baked value before archiving (no full re-prebuild needed):
> `ios/sonoQui/Supporting/Expo.plist` → `EXUpdatesRequestHeaders` → `expo-channel-name`,
> and `android/.../AndroidManifest.xml` → `UPDATES_CONFIGURATION_REQUEST_HEADERS_KEY`
> → set both to `staging`. Leave as `production` for a store archive.

If you use EAS Build instead of local builds, the channel comes from the
profile's `channel` field in `eas.json`; you don't need to set `OTA_CHANNEL`.

## Troubleshooting

### The app never picks up OTA updates

1. The installed binary must have been built with a `channel`. Builds made
   before OTA was added have none and fetch nothing — rebuild + reinstall.
2. Log `Updates.channel`, `Updates.runtimeVersion`, and
   `Updates.checkForUpdateAsync()` from a release build to see what the client
   reports.
3. Hit the manifest directly with the binary's runtimeVersion:
   ```bash
   curl -i "https://ota.sonoqui.pro/api/manifest" \
     -H "expo-channel-name: staging" \
     -H "expo-runtime-version: 1" \
     -H "expo-platform: ios" \
     -H "expo-protocol-version: 1"
   ```
   200 = a bundle exists for that pair; 404 = none published for this runtime.

### `eoas publish` fails with an auth error

Either you're not logged into Expo (`npx eas login` as `micheel93-2`) /
`EXPO_TOKEN` is stale, or the server's `EXPO_ACCESS_TOKEN` in
`/opt/sonoqui/.env` is out of sync. Keep the publisher and server authorised
for the same project.

### `eoas publish` hangs / 403s on the Cloudflare challenge

Cloudflare Bot Fight Mode can 403 datacenter IPs. From a residential/office IP
this is normally fine. If you hit it, pin the host to the origin and skip CF for
the publish only:

```bash
echo "57.131.52.5 ota.sonoqui.pro" | sudo tee -a /etc/hosts
NODE_TLS_REJECT_UNAUTHORIZED=0 npm run ota:publish     # bundles are RSA-signed, so TLS here is defence-in-depth
# remove the /etc/hosts line afterwards
```

### The OTA container is down

```bash
ssh -p 2222 ubuntu@57.131.52.5
docker ps | grep ota-sonoqui
docker logs ota-sonoqui --tail 40
cd /opt/sonoqui && docker compose up -d ota-sonoqui
```

### Rotate the signing key

1. `cd apps/mobile && rm credentials/*.pem`
2. `npx expo-updates codesigning:generate --key-output-directory credentials --certificate-output-directory credentials --certificate-validity-duration-years 10 --certificate-common-name "sonoQui OTA Code Signing"`
3. SCP the new `private-key.pem` + `public-key.pem` to `/opt/sonoqui/ota/keys/`
   (600 on the private one), restart `ota-sonoqui`.
4. `npm run prebuild` (re-bakes the new `certificate.pem`), commit it.
5. **Rebuild + resubmit natively** — field binaries trust the old cert and won't
   verify updates signed by the new key.
